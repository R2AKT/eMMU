## 📖 О проекте

**MEGA-580 OS** — экспериментальная монолитная операционная система, написанная полностью на ассемблере КР580ВМ80А (~3500 строк кода). ОС разработана для работы на советской микроэлектронике серии КР580 и поддерживает современные POSIX-подобные концепции: вытесняющую многозадачность, виртуальную файловую систему, символьные устройства, сигналы и аппаратную защиту памяти.

Ключевая архитектурная особенность — использование собственной сегментно-страничной системы памяти **eMMU rev_2.0** с четырьмя сегментами по 16 КБ и режимом `KERNEL_OVERRIDE` для безопасного доступа ядра (Ring 0) к пользовательской памяти (Ring 3).

---

## ⚠️ Важное примечание о статусе проекта

> **Подсистема системных вызовов (`SYSCALL_TABLE`) находится в стадии активной переработки.**  
> В текущей сборке таблица `SYSCALL_TABLE` содержит только заглушку `syscall_bad_number`. Реализация полного набора POSIX-сисколлов (`sys_exit`, `sys_open`, `sys_read`, `sys_write`, `sys_gettimeofday` и др.) запланирована в ближайших коммитах.
>
> **Вся остальная инфраструктура ядра** (VFS, RTC, NVRAM, TTY, LPT, планировщик, обработчик прерываний) **полностью функциональна, протестирована и готова к интеграции**.

## ✨ Ключевые особенности

### 🏗️ Архитектура ядра
- **Монолитное ядро** с вытесняющей многозадачностью
- **Двухуровневая защита памяти** (Ring 0 / Ring 3) через eMMU rev_2.0
- **Режим KERNEL_OVERRIDE** для атомарного доступа ядра к памяти Ring 3
- **Динамические приоритеты** с aging и насыщением до 255
- **POSIX-подобные сигналы** (16 сигналов, маски SIG_FATAL)
- **SMC-патчинг** (Self-Modifying Code) для адресации портов ISR

### 📁 Виртуальная файловая система (VFS)
- **Таблица коммутации драйверов** `VFS_SWITCH_TABLE` (32 Major × 10 байт)
- **Таблица монтирования устройств** `DEV_MOUNT_TABLE` (16 слотов × 6 байт)
- **Таблица Inode** `VFS_INODE_TABLE` (64 Inode × 16 байт)
- **Глобальная таблица файлов** `SYS_FILE_TABLE` (128 дескрипторов × 10 байт)
- **Локальная таблица FD** в PCB (16 дескрипторов на процесс)
- **Оптимизация хвостового вызова** через трюк `XTHL` в `k_vfs_dispatch`

### 🕐 Подсистема RTC/NVRAM
- **Аппаратные часы реального времени** на базе КР512ВИ1 (аналог DS12885)
- **Двусторонний конвертер эпохи**: `k_rtc_convert_raw_to_unix` ↔ `k_rtc_convert_unix_to_raw`
- **32-битный UNIX Time** (Little-Endian) с диапазоном 01.01.2026 — 07.02.2106
- **Защита от бита UIP**: атомарное чтение в окно 244 мкс
- **NVRAM 50 байт** (адреса 0x0E — 0x3F) с блочными операциями
- **Обработка високосных лет** через таблицу `k_rtc_days_before_month`

### 🖥️ Подсистема TTY
- **До 4-х независимых консолей** (TTY0 — TTY3)
- **Приоритетный автодетект**: КР580ВВ51А + КР580ВИ53, затем NS16550
- **Кольцевые буферы** RX/TX по 64 байта
- **POSIX termios-флаги**: ECHO, ICANON, ISIG, ICRNL, OPOST
- **Управляющие символы**: Ctrl+C, Ctrl+D, Ctrl+U, Backspace, Ctrl+Q/S

### 🖨️ Подсистема LPT
- **До 2-х параллельных портов** (LPT0, LPT1)
- **Приоритетный автодетект**: Intel 82C11, затем КР580ВВ55А
- **FIFO-буфер отложенной записи** 48 байт
- **Асинхронный Handshake Centronics** через BSR-команды РРУ
- **32-битный счётчик** выведенных байт `O_LP_BYTES_WRITTEN`

### Двухуровневая система диспетчеризации

```
Ring 3: write(fd, buf, len)
    │
    ▼
sys_call_entry (RST 6)
    │
    ▼
sys_write
    │
    ▼
k_vfs_get_file_struct_address(fd) → HL = адрес Inode
    │
    ▼

    │
    ▼

    │
    ▼

    │
    ▼
```

## 🕐 Подсистема RTC/NVRAM

### Полный двусторонний стек конвертации

```
k_rtc_raw_sec..year  ←→  k_rtc_convert_raw_to_unix  ←→  SYS_UNIX_TIME (32 bit)
                            k_rtc_convert_unix_to_raw
```

### Ключевые особенности

- **Защита от бита UIP**: Чтение регистров только в безопасное окно 244 мкс
- **Тест через Регистр B**: Исключение ложных срабатываний от динамического бита UIP
- **Оптимизированная арифметика**: Умножение дней на 86400 через разложение `65536 + 20864`
- **Таблица `k_rtc_days_before_month`** в формате `.dw` (16 бит) для поддержки октября/ноября/декабря
- **Корректная обработка високосных лет**: `(год % 4 == 0) && (месяц > 2)`

### VFS-мосты

| Устройство | Major | Minor | Функции |
|------------|-------|-------|---------|
| `/dev/rtc` | 8 | 1 | `rtc_read_bridge`, `rtc_write_bridge` |
| `/dev/nvram` | 9 | 0 | `nvram_read_bridge`, `nvram_write_bridge` |

Все мосты используют атомарное управление `KERNEL_OVERRIDE` через порт `0x76` для безопасного доступа к памяти Ring 3.

## 🔄 Планировщик задач

### Динамические приоритеты с aging

```
Формула приоритета: DYNAMIC = STATIC + NICE + CPU_TICKS
Насыщение: min(255, DYNAMIC)
```

- **Штрафной счётчик**: `O_CPU_TICKS` увеличивается при каждом кванте выполнения
- **Aging**: `O_CPU_TICKS--` для всех процессов, кроме `ST_ZOMBIE`
- **Выбор кандидата**: Линейный скан 32 слотов `PCB_TABLE` с поиском минимума
- **Сброс тиков**: `O_CPU_TICKS = 0` для запускаемого процесса

### Обработка будильников (POSIX `alarm`)

Каждый квант (20 мс) ISR сканирует `O_ALARM_TICKS` всех процессов. При достижении нуля — доставка сигнала `SIGALRM` в старший байт маски `O_SIG_PENDING`.

## 📡 POSIX-сигналы

### 16-битная маска сигналов

**Младший байт** (`O_SIG_PENDING + 0`): сигналы 1 — 8
- `SIG_HUP` (1), `SIG_INT` (2), `SIG_QUIT` (3), `SIG_ILL` (4)
- `SIG_TRAP` (5), `SIG_ABRT` (6), `SIG_BUS` (7), `SIG_FPE` (8)

**Старший байт** (`O_SIG_PENDING + 1`): сигналы 9 — 16
- `SIG_KILL` (9), `SIG_USR1` (10), `SIG_SEGV` (11), `SIG_USR2` (12)
- `SIG_PIPE` (13), `SIG_ALRM` (14), `SIG_TERM` (15), `SIG_CHLD` (16)

### Комбинированные маски смертельных сигналов

```asm
SIG_FATAL_MASK_L = SIGHUP OR SIGINT OR SIGQUIT OR SIGILL OR SIGABRT OR SIGBUS OR SIGFPE
SIG_FATAL_MASK_H = SIGKILL OR SIGSEGV OR SIGTERM
```

## 🛡️ Безопасность и защита памяти

### eMMU rev_2.0

- **4 сегмента** по 16 КБ (`EMMU_PAGE_SIZE = 4000H`)
- **Ring 0 / Ring 3** разделение через порт `0x76` (`EMMU_OVERRIDE`)
- **Порты управления**: `0x70..0x73` (регистрация сегментов), `0x75` (отключение ПЗУ)

### Обработчик нарушения привилегий (`hw_privilege_violation_entry`)

При попытке Ring 3 выполнить `IN`/`OUT`/`EI`/`DI`:
1. Перевод процесса в статус `ST_ZOMBIE`
2. Освобождение физических страниц через `k_free_physical_page`
3. Закрытие всех открытых файлов
4. Переключение контекста на живой процесс через `switch_mmu_context`
